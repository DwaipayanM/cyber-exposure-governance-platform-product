# Cyber Exposure Governance Platform (CEGP) - Project Submission Overview

## 1. Project Overview & Motivation
### 1.1 What Initiated This Project
In modern enterprise cybersecurity, organizations are overwhelmed by the volume of vulnerability alerts generated by automated scanning tools. Standard vulnerability management programs prioritize patching based on technical severity ratings (specifically the Common Vulnerability Scoring System, or CVSS). 

This traditional approach suffers from several key limitations:
* **Absence of Real-world Threat Context**: High-severity technical vulnerabilities are often not actively exploited in the wild, while lower-severity ones may be actively targeted by cybercriminals.
* **Lack of Business Context**: An internal test server is treated with the same urgency as a customer-facing production database.
* **Absence of Live Detection Signal**: Traditional scanners do not correlate network traffic alerts with host vulnerabilities, meaning systems currently under active attack are not flagged immediately.
* **Lack of Governance & Verification**: Auditing remediation timelines and verifying report integrity to satisfy regulatory compliance (like SOC 2 or ISO 27001) remains manual and error-prone.

The **Cyber Exposure Governance Platform (CEGP)** was initiated as a decision-support product prototype. It bridges the gap between raw vulnerability finding lists and strategic business risk management. It correlates threat intelligence, asset criticality, network zone placement, live security alert activity, and data privacy impact into a single, explainable priority score.

## 2. Core Capabilities
The platform supports a robust risk management lifecycle:
1. **Multi-Source Data Ingestion**: Standardizes vulnerability lists from different network scanners into a unified format.
2. **Threat Intelligence Enrichment**: Correlates vulnerabilities with live databases tracking known exploited vulnerabilities (CISA KEV) and exploit probabilities (EPSS).
3. **Business Context Correlation**: Enriches alerts with asset details, business processes, and data sensitivity.
4. **Network Exposure Mapping**: Factors in network zones (such as public internet versus internal segments), open ports, and firewall access policies.
5. **Security Alert Correlation**: Dynamically matches network security events (IDS/IPS alerts) against vulnerable hosts to identify systems under active target attempts.
6. **Privacy Risk Quantification**: Scores systems based on the presence of personal data (PII), encryption status, and regulatory impact.
7. **Configurable Risk Policy**: Allows security leaders to adjust weights, priority thresholds, and remediation SLAs according to corporate risk appetite.
8. **Automated Playbooks & SLA Tracking**: Assigns due dates based on priority, raises escalation warnings for breached timelines, and generates remediation instructions.
9. **What-If Risk Simulation**: Simulates patch strategies and security controls to show predicted reductions in overall corporate risk before allocating engineering resources.
10. **Tamper-Evident Report Signing**: Secures exported reports with cryptographic signatures to ensure audit trails cannot be edited after export.
11. **Attributed Audit Logging**: Records all analyst actions to maintain an authoritative history for compliance reviews.

## 3. High-Level Implementation Steps
The platform processes data through a structured pipeline. Each phase adds a layer of intelligence to determine the final governance actions:

### Phase 1: Data Ingestion & Sanitization
The platform reads raw CSV or Excel data. It cleans column names, ensures required fields are present, maps optional fields to standard templates, and validates input formatting. Non-blocking data-quality issues are caught and saved for review, allowing processing to continue.

### Phase 2: Threat Intelligence Contextualization
Vulnerabilities are enriched with external intelligence:
* Checks whether the vulnerability is actively exploited in the wild (CISA KEV).
* Analyzes the likelihood of exploitation within the next 30 days (EPSS).
* Imports the default technical severity (CVSS).

### Phase 3: Asset & Network Context Merging
The system merges the vulnerability entries with the organization’s asset inventory:
* **Business Impact**: Assigns weight based on asset criticality, whether it is a production environment, and its relevance to core business processes.
* **Network Exposure**: Examines the asset’s host location (e.g., Internet vs. Internal), open ports, and whether a VPN is required to access it.

### Phase 4: Active Threat & Privacy Mapping
* **IDS/IPS Correlation**: Checks if there are live network intrusion events directed at the vulnerable asset. Assets under active target attempts are flagged for expedited patching.
* **Privacy Evaluation**: Identifies if the asset holds sensitive personal data and checks if that data is unencrypted, which increases compliance exposure.

### Phase 5: Score Calculation & SLA Assignment
The platform calculates a consolidated 0-100 cyber exposure score. Based on this score, vulnerabilities are categorized into Low, Medium, High, or Critical priorities. Remediation SLAs (such as 7 days for Critical and 30 days for Medium) are established based on the date of detection.

### Phase 6: Playbook Generation & Remediation Actions
Remediation playbooks are created for each exposure. These include:
* **Primary remediation action** (e.g., patching or software updates).
* **Compensating controls** (e.g., restricting network access or enabling encryption).
* **Validation steps** to verify the fix.

### Phase 7: Export & Integrity Verification
Finally, the platform sanitizes exported spreadsheets against common injection risks and generates a cryptographic signature. Administrators can use the platform's verification tool to upload any report and check if its contents have been altered.

## 4. How to Execute & Run the Project Locally
The application is built on Streamlit (frontend UI) and Pandas (engine processing).

### Step 4.1: Open PowerShell / Terminal
Open your terminal and navigate to the project directory:
```powershell
cd C:\cyber-exposure-governance-platform-product
```

### Step 4.2: Setup Virtual Environment & Install Dependencies
Initialize the Python virtual environment and install the required libraries:
```powershell
# Create virtual environment using the Python launcher
py -m venv .venv

# Activate the virtual environment
.\.venv\Scripts\activate

# Install requirements
pip install -r requirements.txt
```

### Step 4.3: Start the Application
Run the Streamlit server to start the web interface:
```powershell
streamlit run app.py
```
After executing this, the application will automatically open in your default browser, or you can access it at:
👉 **[http://localhost:8501](http://localhost:8501)**

## 5. Project Outcomes & Benefits
The platform changes how security teams and auditors view vulnerability posture:

* **Efficient Prioritization**: Cuts down large lists of hundreds of technical vulnerabilities to a focused list of high-priority exposures that have real-world exploit paths.
* **What-if Scenario Planning**: Enables risk managers to simulate patching the top riskiest assets or isolating public-facing servers. The dashboard instantly shows the potential percentage reduction in corporate risk.
* **Audit Readiness**: Provides signed reports and a secure audit log showing who performed each assessment and when. This can be presented directly to regulatory auditors.
* **Actionable Remediation**: Teams receive clear playbooks and SLA countdowns instead of generic technical notes, accelerating the patch cycle.
